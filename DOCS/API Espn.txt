Esses endpoints da ESPN funcionam mesmo para obter dados em tempo (quase) real de NFL, MLB, NHL, college, NBA, soccer etc., mas são não-oficiais e podem mudar ou ser bloqueados sem aviso.
​

Avisos importantes
API não documentada / não suportada pela ESPN; foi descoberta via reverse engineering do próprio site/apps.
​

Uso em produção para produto comercial é arriscado: endpoints podem mudar, rate limits podem apertar, e pode violar termos de uso se abusar.
​

Sempre coloca um layer no teu sistema (proxy interno) para facilitar troca de fornecedor depois.
​

Como integrar no teu sistema (alto nível)
Sugestão para o teu stack de betting/analytics:

Cria um serviço (FastAPI, por exemplo) que exponha endpoints internos como:

/espn/nfl/scoreboard?date=YYYY-MM-DD → faz proxy para o site.api.espn.com e normaliza o JSON.
​

/espn/soccer/:league/scoreboard?date=YYYY-MM-DD

Implementa:

Cache (Redis) de 5–15 segundos para live e 1–5 minutos para dados históricos.
​

Normalização de IDs (liga, equipa, jogo) para bater com teu modelo interno / casas de apostas.
​

Tratamento de falhas (timeout, mudanças de schema).

Dá para ter tudo isso: jogos ao vivo com tempo/estado, jogos de hoje e agenda por intervalo de datas (ex.: segunda–domingo) usando os mesmos endpoints, combinando scoreboard, summary e parâmetros de data.
​

Jogos do dia e ao vivo (com tempo e estado)
Para cada esporte/liga (ex.: futebol EPL eng.1):

Scoreboard do dia (inclui jogos passados, em andamento e futuros daquele dia):

https://site.api.espn.com/apis/site/v2/sports/soccer/eng.1/scoreboard → sem dates assume hoje.
​

Live status dentro do JSON:

Em events[*].status tens campos como type.name (STATUS_IN_PROGRESS, STATUS_FINAL, STATUS_SCHEDULED), type.state (in, post, pre) e displayClock (45'+2, 73:15, etc.).
​

Em futebol, o placar e tempo vêm em competitions[0].status e competitions[0].competitors[*].score.
​

Para estatísticas mais detalhadas de um jogo específico, usas o event.id do scoreboard:

https://site.api.espn.com/apis/site/v2/sports/soccer/eng.1/summary?event={eventId} → boxscore, stats, lineups, etc., atualizando durante o jogo.
​

Mesma ideia vale para NBA, vólei (quando suportado) e ténis, mudando o caminho do esporte/liga.
​

Jogos da semana / intervalo de datas
Os scoreboards aceitam:

dates=YYYYMMDD → um dia.
​

dates=YYYYMMDD-YYYYMMDD → intervalo de datas, por exemplo, segunda a domingo:

https://site.api.espn.com/apis/site/v2/sports/soccer/eng.1/scoreboard?dates=20251222-20251228.
​

Isso retorna todos os jogos da liga nesse intervalo (já jogados, ao vivo e futuros), e cada evento traz date e status para filtrar no teu sistema por:

“já começou”, “ao vivo”, “a começar em X minutos”, “futuro na semana”, etc.
​

Para toda a bola num intervalo (todas ligas soccer que a ESPN cobre):

https://site.api.espn.com/apis/site/v2/sports/soccer/all/scoreboard?dates=20251222-20251228.
​

Estatísticas in-game
Fluxo típico para ter live + stats:

Chama o scoreboard da liga/dia ou intervalo (scoreboard).
​

Para cada jogo que te interessa, guarda event.id.
​

Periodicamente (por exemplo, a cada 10–20 s só para jogos STATUS_IN_PROGRESS), chama:

.../summary?event={eventId} para carregar:

gols, remates, posse (quando disponível), cartões, substituições, etc.
​

Para ténis / outros esportes, o live detalhado por vezes vem mais pela sports.core.api.espn.com (events, competitions, statistics) mas o padrão scoreboard + summary continua o mesmo.
​

Como montar no teu sistema (ideia prática)
Endpoints internos que podes criar:

/espn/soccer/{league}/games?date=YYYY-MM-DD → faz proxy para scoreboard?dates=YYYYMMDD.
​

/espn/soccer/{league}/games?from=YYYY-MM-DD&to=YYYY-MM-DD → mapeia para dates=YYYYMMDD-YYYYMMDD.
​

/espn/event/{eventId}/summary → proxy para .../summary?event={eventId}.
​

No teu banco, salva:

event_id, home_team_id, away_team_id, status.state, status.type.name, start_time, league, + stats que for usar nos modelos de odds/algoritmos.
